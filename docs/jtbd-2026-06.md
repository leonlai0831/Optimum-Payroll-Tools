## 一、运营者跑发薪(Leon / 管理层)

**情境**:每月 1 号上月 KPI 数据刚到 → 要在银行转账截止前,在**手机上**,把 ~180 个自由教练 +
全职员工的薪水**算对、付对**。
**现有 / 被替代方案**:`FREELANCER_CALCULATOR.xlsx`、手工银行转账表、Payment Summary xlsx。

### 1. 客户任务(Jobs)

**🔧 功能性**
- 把上月学生数据 → 算出每位教练 **KPI 分数 + 奖金**
- 算 ~180 自由教练月薪(时薪 × 工时 + commitment + attendance,跨公司分账 OT/OTG/PJ/QSM/KM)
- 算全职津贴,并把上月数额 + payee 资料**自动带过来**
- 处理真实杂项:**补交、缺勤、CC 职位、多账号合并**
- **一键导出银行转账文件**(每个发薪主体一个 sheet:IC / 银行码 / 账号 / 金额)
- 每月结果**存云端、可追溯、可对账、能看趋势**

**👔 社会性**
- 在合伙人 / 老板面前:精准、系统化,不像「靠一个 Excel 撑着」
- 对员工:发薪公平透明,教练信得过
- 对外部开发者 / 会计:这是一套真系统

**❤️ 情感性**
- 确信**没发错钱**(发薪算错 = 真金白银的事故)
- 月底**不熬夜对 Excel、不焦虑**
- 对「系统会自动算对」有掌控感与信任

### 2. 痛点(Pains)

**挑战**
- 数据到得晚(月 P 的数据 P+1 月 1 号才到),时间窗口紧
- 同一教练多个账号(`CK [BK]` / `CK [PK]`)要合并才算得对
- 多公司分账规则复杂

**成本(时间 / 金钱 / 精力)**
- 以前几百人手算 Excel,几小时起跳,易错
- ⚠️ **一次性导入 ~180 个 freelancer 档案还没做**(待办)
- payee 银行资料补全是体力活
- **每月手工录入工时**(allowance 的 teachingRows、freelancer 的 fixed/replaced)

**常见错误**
- 重复保存覆盖记录(已有 `ConfirmModal` 防护)
- 名字合并把两个人当成一个(或反之)
- 漏掉补交 / 缺勤行
- center target 配错 → 主管的组分目标悄悄翻车(已做 deterministic 防护)

**未解决问题**
- ⚠️ 真机视觉 QA 还没在真手机上跑过(待办)
- ⚠️ freelancer v1 **没有期间锁**

### 3. 收益(Gains)

- **超出期望**:引擎能 **byte-for-byte 复现 Excel**(5 月 22/22 抽样 delta RM 0.00,总额 RM 245,759.30 对得上);AI 自动合并账号、自动 carry-over
- **节省**:几小时 → 几分钟;不用再养 Excel
- **采纳因素**:默认完全复现 v11.1,**旧记录可比**;手机可用;云端共享
- **生活改善**:月底准时下班、少焦虑、出问题有审计可查

### 🎯 最痛的一条
**「一键把 ~180 人从 Excel 迁进系统,并跑通一次真实 6 月发薪」** —— 即 roadmap 的
**P0:payee 一键导入 → 6 月并行跑 → payee 完整度**。其它都次于它。

## 二、教练和自由教练(被发薪 / 被考核的人)

**情境**:拿到月薪那天 → 想搞懂我的钱是怎么算的,并相信它没错。
**现有方案**:WhatsApp 私信问 admin、纸质教案、单纯「信任老板的 Excel」。

> ⚠️ **关键事实**:今天他们**几乎都是间接用户**。所以本段很多是「潜在需求」,不是「现有行为」。
> 运营者已决定要把教练变成直接用户(见末尾「打卡系统」),本段因此从「想象」升级为「即将落地」。

### 1. 客户任务(Jobs)

**🔧 功能性**
- 看懂自己这个月薪资 / 奖金**怎么算出来的**(时薪、工时、commitment、attendance)
- 知道自己的 **KPI 分数、等级、排行榜位置**
- 看到 / 确认自己的**考核反馈**
- 提交自己的**教案**(`edit_lesson_plans`)
- 看自己的档案(My Profile,跨类目永远可达)
- ⚠️(新)**记录自己每月教了什么课、什么 level、多少小时**

**👔 社会性**:在主管 / 同事面前显得专业、可靠、上进
**❤️ 情感性**:确信「我没被少算」;对发薪**公平、透明**有安全感;升级 / 通过时的成就感

### 2. 痛点(Pains)

- **挑战**:薪资算法是黑盒,只能问 admin;补交 / 缺勤怎么影响我说不清
- **成本**:每月为「这数对吗」来回 WhatsApp,双方都耗时
- **常见错误**:⚠️ 自己记错工时 / 缺勤,跟系统对不上,产生争议
- **未解决问题**:⚠️ **没有自助查薪 / 查 KPI 的入口**;⚠️ 工时靠口头 / 纸面回报给 admin

### 3. 收益(Gains)

- **超出期望**:手机上一目了然「我的薪资明细 + KPI + 等级」
- **节省**:省掉来回问 admin 的沟通成本
- **采纳因素**:数字跟到手的钱一致、解释清楚 → 信任
- **生活改善**:对收入有掌控感、少焦虑、知道怎么提升 KPI 拿更高奖金

### 🎯 最痛的一条
**薪资 / KPI 的「黑盒」+ 缺自助入口 + 工时回报靠人肉**。打卡系统正是冲这条来的。

## 三、主管 / 评审(泳池主管)

**情境**:月末 → 要复核手下教练的 KPI、做完考核、批 / 退教案。
**现有方案**:系统里的审批流 + 当面 / WhatsApp 带教。

### 1. 客户任务(Jobs)

**🔧 功能性**
- 复核手下教练的 **KPI** 结果
- 填**教练考核**(KPI 里 manual 的「Mgmt Assessment」分来自这里)
- **批准 / 打回教案**(`review_lesson_plans`)
- 给**组分 / 中心工时**输入(Pool Supervisor 的 group score 依赖)

**👔 社会性**:被看作公平、有判断力的带教者
**❤️ 情感性**:确信复核**一致、不出错**;对团队表现有掌控

### 2. 痛点(Pains)

- **挑战**:⚠️ 要凑齐多种输入(工时、考核分、组分)才能让月度数字成立,易卡在「等某人交数据」
- **成本**:⚠️ 手机上做审批 / 考核若不顺手就费劲
- **常见错误**:漏批某份教案、考核分填错档
- **未解决问题**:⚠️ 没有清爽的「待我复核」队列总览(需确认现状)

### 3. 收益(Gains)

- **超出期望**:打开就看到「待我复核 / 待我批」清单
- **节省**:批量、手机友好的审批
- **采纳因素**:与 KPI / 发薪打通(我填的考核分直接进奖金)
- **生活改善**:月末复核不再是负担

### 🎯 最痛的一条
**输入的「凑齐」+ 移动端审批顺手度**。优先级低于运营者那条,属打磨。

## 跨三类的综合结论 & 优先级

**一句话**:三类人里**只有「运营者跑发薪」是现在真正烧手的**;另外两类是战略方向题,优先级更低。
这跟 roadmap 的 P0 一致。

1. **P0(立刻)**:运营者 → payee 一键导入 ~180 人 → 6 月真实并行跑 → 对账。**上线阻塞,别的让路。**
2. **P1(打磨)**:真机视觉 QA、错误日志 / 可观测性(PR #148 已带)。
3. **P2(战略,已定方向)**:让**教练成为直接用户** → 通过**打卡系统**(见下)。

## 浮现的新倡议:打卡系统(Clock-in / Lesson Log)

**愿景(运营者原话)**:做一个打卡系统,记录教练整月教了什么课、什么 level,从而**自动 update
Staff Allowance、KPI Bonus、Freelancer Payment**。

### 三引擎数据匹配度(关键)

| 引擎 | 真正吃的数据 | 打卡能否原生产出 |
| --- | --- | --- |
| **Staff Allowance** | `teachingRows`(每中心 × 课型 normal/ys/precomp 的**小时**)+ `opHours`/`leaveHours` | ✅ **原生命中** —— 打卡天然产出每节课的中心 / 课型 / 时长 / 出勤 |
| **Freelancer Payment** | 每中心 `fixedHours` / `replacedHours` + 缺勤 | ✅ **原生命中** —— 同上;还能区分固定 / 替补 |
| **KPI Bonus** | 学生进度指标(升级率、黑色率、留存、效率…),即**每个学生的成绩结果** | ⚠️ **不命中核心** —— 「老师教了什么 level」≠「学生升了几个 level」;KPI 仍需现有学生数据管道。打卡只能补 attendance/efficiency 类输入 |

**结论**:打卡系统本质是一套**「工时 / 出勤的单一真实来源」**。它能**直接自动化 Allowance + Freelancer
的工时录入**(消灭当前最大的手工录入痛点),但**不能自动算 KPI bonus 的核心进度指标** —— 那来自学生
成绩,不来自教师自报的课。所以「自动 update 三个」要拆开讲:**工时类全自动,学生成绩类仍走原管道**。

### 最窄有价值的 wedge(建议 v1)

**打卡 → 自动填好 Allowance + Freelancer 的工时**,让教练成为直接用户;KPI 维持现有学生数据管道。
干净、可上线、拿下绝大部分价值,且避开「想从打卡里凑出学生进度」的陷阱。

### 待定问题(下一步要先答)

1. **打卡颗粒度**:是「每节课(level / 课型 / 中心 / 时长 / 是否出勤)」?是否也记**哪些学生到课**
   (→ 能补 KPI 的 Efficiency / Attended)还是只记课本身?
2. **替补 vs 固定**(freelancer):教练能否自报,还是要主管确认?(信任 / 核验)
3. **KPI 边界**:确认 KPI 的学生进度数据**仍来自现有月度学生 ingest**,打卡不替代它?
4. **核验流**:打卡是否需主管 / admin 审批后才进发薪?(防自报虚高工时)
